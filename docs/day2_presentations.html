Classes, Objects, & Constructors

The fundamental building blocks of Object-Oriented Programming: defining data structures and their initial state.

Classes Objects Instantiation Constructors `this` keyword

**Trainer Note:** "We'll kick off with the basics: how we define what an entity is (`Class`) and then bring it to life (`Object`). We'll also cover how objects are 'born' with Constructors."

Core Concepts

Classes: The Blueprints

What is a Class?

  • A blueprint or template for creating objects.
  • Defines the characteristics (attributes/fields) and behaviors (methods) that its objects will have.
  • Does NOT consume memory for data when defined; it's a logical construct.
  • Example: A `Car` class defines what a car *is* (color, model, speed) and what it *does* (start, stop, accelerate).

What is an Object?

  • A real-world entity or instance of a class.
  • Created from a class blueprint using the `new` keyword.
  • Each object has its own unique set of attribute values and can perform the behaviors defined by its class.
  • Example: `myRedCar` is an object of the `Car` class, with `color="Red"`, `model="Sedan"`, `speed=0`.
💡
Think of a class as a cookie cutter, and objects as the actual cookies. All cookies are made from the same cutter, but each is a distinct cookie.

Defining a Simple Class:

Car.java
public class Car { String model; String color; int speed; // Method to display car details public void displayCarInfo() { System.out.println("Model: " + model + ", Color: " + color + ", Speed: " + speed + " km/h"); } }

Creating Objects (Instantiation):

Main.java
public class Main { public static void main(String[] args) { // Creating the first Car object Car car1 = new Car(); car1.model = "Toyota"; car1.color = "Blue"; car1.speed = 100; car1.displayCarInfo(); // Creating another Car object Car car2 = new Car(); car2.model = "Honda"; car2.color = "Red"; car2.speed = 120; car2.displayCarInfo(); } }

**Trainer Note:** "Notice how `car1` and `car2` are separate entities, each with its own `model`, `color`, and `speed`."

Object Initialization

Constructors & the `this` Keyword

What is a Constructor?

  • A special type of method used to initialize newly created objects.
  • It has the same name as the class and no return type (not even `void`).
  • Called automatically when an object is created using the `new` keyword.
  • If you don't define any constructor, Java provides a default (no-argument) constructor.

Types of Constructors:

  • Default Constructor: Provided by Java if none are explicitly defined.
  • No-argument Constructor: Explicitly defined constructor that takes no parameters.
  • Parameterized Constructor: Takes one or more parameters to initialize object fields.
  • Copy Constructor: (Specialized) Used to create a new object as a copy of an existing object.

The `this` Keyword:

  • A reference to the current object.
  • Used to distinguish between instance variables and local/parameter variables when they have the same name (shadowing).
  • Can be used to invoke the current class's constructor (constructor chaining: `this(...)`).

Example: Constructors & `this`

CarWithConstructors.java
public class Car { String model; String color; int speed; // No-argument constructor public Car() { this("Generic", "White", 0); // Constructor chaining System.out.println("Default car created."); } // Parameterized constructor (main) public Car(String model, String color, int speed) { this.model = model; // 'this' to refer to instance variable this.color = color; this.speed = speed; } // Copy constructor (creates a new Car from an existing one) public Car(Car otherCar) { this(otherCar.model, otherCar.color, otherCar.speed); // Uses another constructor System.out.println("Car copied."); } // toString() method for easy object printing @Override public String toString() { return "Car [model=" + model + ", color=" + color + ", speed=" + speed + "]"; } public static void main(String[] args) { Car myCar = new Car("Honda Civic", "Black", 150); System.out.println(myCar); Car defaultCar = new Car(); System.out.println(defaultCar); Car copiedCar = new Car(myCar); System.out.println(copiedCar); } }

**Trainer Note:** "Observe how the `this` keyword resolves ambiguity between local and instance variables, and how `this(...)` allows one constructor to call another, preventing code duplication."

Access Specifiers, Static & `this` Reference

Controlling visibility and managing class-level vs. object-level members.

`public` `private` `protected` Default (Package) `static` keyword `this` reference

**Trainer Note:** "Now that we know how to build objects, let's learn how to protect their internal state and how to define behaviors that belong to the class itself, not just an individual object."

Controlling Visibility

Access Specifiers in Java

Purpose of Access Specifiers:

  • Determine the visibility and accessibility of classes, fields, methods, and constructors.
  • Crucial for encapsulation – controlling direct access to an object's internal state.
  • Helps in protecting data integrity and simplifying code maintenance by defining clear interfaces.

The Four Access Levels:

Modifier Same Class Same Package Subclass (different package) Other Package
private ✅ ❌ ❌ ❌
(no modifier)
(Default / Package-Private)		 ✅ ✅ ❌ ❌
protected ✅ ✅ ✅ ❌
public ✅ ✅ ✅ ✅

Best Practices:

  • Encapsulate: Make fields `private`. Provide `public` getters/setters if external access is needed. This allows you to add validation logic to setters.
  • Package-Private (Default): Use for utility classes or members strictly internal to a package.
  • Protected: Reserve for members that subclasses in other packages need to access, or for framework-level extensibility.
  • Public: For methods that define the public API of your class. Use sparingly for fields.

Example: Access Specifiers in Action

AccessExample.java
package com.example.model; public class Student { private String name; // Accessible only within Student class int id; // Default: package-private protected String grade; // Accessible in package & subclasses public String university; // Accessible everywhere public Student(String name, int id) { this.name = name; this.id = id; } public String getName() { // Public getter for private field return name; } } // In another package, only public members are directly accessible.

**Trainer Note:** "Understanding access specifiers is key to writing robust, maintainable, and secure Java applications. It directly supports the encapsulation principle."

Class-Level Members

The `static` Keyword

What `static` Means:

  • The `static` keyword makes a member (field, method, or nested class) belong to the class itself, rather than to any specific instance of the class.
  • Static Fields: Shared by all instances of the class. There's only one copy in memory, regardless of how many objects are created.
  • Static Methods: Can be called directly on the class name (e.g., `ClassName.staticMethod()`) without creating an object. They can only access static members (fields and methods) directly.
  • Often used for utility methods (like `Math.random()`) or constants.

Why use `static`?

  • Memory Management: Saves memory as static members are loaded once.
  • Utility Functions: Ideal for methods that don't depend on object-specific data.
  • Constants: `public static final` fields are common for global constants.
  • Single Point of Truth: For data that should be consistent across all instances (e.g., a counter for created objects).
💡
A static method cannot refer to `this` or `super` keywords, as they are used to refer to instance-specific entities.

Example: Static Members

StudentCounter.java
public class Student { String name; private static int studentCount = 0; // Static field public Student(String name) { this.name = name; studentCount++; // Increments the shared count } public static int getStudentCount() { // Static method return studentCount; } public void displayInfo() { System.out.println("Student Name: " + name + ", Total Students: " + studentCount); } public static void main(String[] args) { Student s1 = new Student("Alice"); Student s2 = new Student("Bob"); Student s3 = new Student("Charlie"); s1.displayInfo(); s2.displayInfo(); System.out.println("Total students created: " + Student.getStudentCount()); } }

**Trainer Note:** "Notice how `studentCount` reflects the total number of students created, not just the count for a single `s1` or `s2`."

Data Manipulation

Wrapper Classes & Type Casting

Wrapper Classes: Bridging Primitives & Objects

  • Java is object-oriented, but it includes primitive data types (`int`, `char`, `boolean`, etc.) for performance.
  • Wrapper classes (`Integer`, `Character`, `Boolean`, etc.) provide object representations for these primitives.
  • Why use them?
    • To store primitive values in collections (e.g., `ArrayList`).
    • To use utility methods (e.g., `Integer.parseInt()`).
    • For methods that accept `Object` types.
  • Autoboxing & Unboxing: Java automatically converts between primitives and their wrappers.
    • Autoboxing: `int` to `Integer`.
    • Unboxing: `Integer` to `int`.

Type Casting: Converting Data Types

  • Used to convert a value from one data type to another.
  • Widening (Implicit) Casting: Automatic conversion from a smaller type to a larger type (e.g., `int` to `double`). No data loss.
  • Narrowing (Explicit) Casting: Manual conversion from a larger type to a smaller type (e.g., `double` to `int`). Requires explicit cast operator `()` and may result in data loss or precision loss.
  • Object Casting: Converting between related object types (e.g., `Animal` to `Dog`). Requires careful use of `instanceof` to avoid `ClassCastException`.

Example: Wrapper Classes

WrapperExample.java
public class WrapperExample { public static void main(String[] args) { // Autoboxing: primitive int to Wrapper Integer int primitiveInt = 100; Integer wrapperInt = primitiveInt; // Autoboxing System.out.println("Wrapper Integer: " + wrapperInt); // Unboxing: Wrapper Integer to primitive int int anotherInt = wrapperInt; // Unboxing System.out.println("Primitive int: " + anotherInt); // Using wrapper class utility method String numberString = "123"; int parsedInt = Integer.parseInt(numberString); System.out.println("Parsed int: " + parsedInt); } }

Example: Type Casting

CastingExample.java
public class CastingExample { public static void main(String[] args) { // Widening (Implicit) Casting int myInt = 10; double myDouble = myInt; // int to double, automatic System.out.println("Int to Double: " + myDouble); // Narrowing (Explicit) Casting double anotherDouble = 9.78; int anotherInt = (int) anotherDouble; // double to int, explicit, data loss System.out.println("Double to Int: " + anotherInt); // Object Casting (Upcasting/Downcasting) Object obj = "Hello"; // Upcasting (implicit) String str = (String) obj; // Downcasting (explicit) System.out.println("Object to String: " + str); } }

**Trainer Note:** "Wrapper classes provide a crucial bridge in Java's hybrid object-oriented and primitive type system. Casting allows us to control type conversions, but always be mindful of potential data loss with narrowing casts."

Data Manipulation

String Handling in Java

Strings: Immutable Sequences of Characters

  • `java.lang.String` is one of the most fundamental classes in Java.
  • Immutability: Once a `String` object is created, its contents cannot be changed. Any operation that appears to modify a `String` actually creates a new `String` object.
    • Example: `String s = "hello"; s = s + " world";` creates two `String` objects.
  • This immutability makes `String` safe for multi-threading and suitable for keys in `HashMap`.

`StringBuilder` & `StringBuffer`: Mutable Alternatives

  • For scenarios requiring frequent string modifications (e.g., building a long string in a loop), `String` concatenation (`+`) is inefficient due to its immutability.
  • `StringBuilder`: A mutable sequence of characters. Ideal for single-threaded environments where performance is critical. Appends and inserts modify the object in place.
  • `StringBuffer`: Similar to `StringBuilder`, but `synchronized` (thread-safe). Slower than `StringBuilder` due to the overhead of synchronization, but safer for multi-threaded access.
  • Use `append()`, `insert()`, `delete()`, `reverse()` methods.

Example: String Immutability vs. StringBuilder

StringVsBuilder.java
public class StringVsBuilder { public static void main(String[] args) { // Demonstrating String Immutability String s1 = "Java"; String s2 = s1; // s2 refers to the same "Java" object s1 = s1 + " is Great"; // s1 now refers to a *new* String object System.out.println("s1: " + s1); // Java is Great System.out.println("s2: " + s2); // Java (original object unchanged) // Demonstrating StringBuilder Mutability StringBuilder sb1 = new StringBuilder("Hello"); StringBuilder sb2 = sb1; // sb2 refers to the same StringBuilder object sb1.append(" World"); // sb1 modifies the existing object System.out.println("sb1: " + sb1); // Hello World System.out.println("sb2: " + sb2); // Hello World (same object modified) // Performance difference (conceptual) long startTime = System.nanoTime(); String tempString = ""; for (int i = 0; i < 1000; i++) { tempString += "a"; // Creates 1000 new String objects } long endTime = System.nanoTime(); System.out.println("String concat time: " + (endTime - startTime) + " ns"); startTime = System.nanoTime(); StringBuilder tempBuilder = new StringBuilder(); for (int i = 0; i < 1000; i++) { tempBuilder.append("a"); // Modifies single StringBuilder object } tempBuilder.toString(); // Convert to String once endTime = System.nanoTime(); System.out.println("StringBuilder append time: " + (endTime - startTime) + " ns"); } }

**Trainer Note:** "The immutability of `String` is a fundamental concept. It's why `StringBuilder` and `StringBuffer` exist. The performance difference with frequent modifications is significant."

Inheritance & Polymorphism

Building hierarchies and writing flexible, extensible code through abstraction.

`extends` keyword `super` keyword Method Overriding Method Overloading `final` keyword `is-a` relationship

**Trainer Note:** "Welcome back! Now that we've covered the basics of defining objects, let's explore how we can build relationships between them. This is where the true power of OOP for code reuse and extensibility really shines."

Core Concept

Inheritance: Building on Existing Code

Hierarchical Relationship:

  • Parent Class (Superclass/Base class): The class whose features are inherited.
  • Child Class (Subclass/Derived class): The class that inherits features from another class.
  • A subclass `extends` a superclass, establishing an "is-a" relationship (e.g., a Dog *is a* kind of Animal).

Benefits:

  • Code Reusability: Common code is written once in the superclass and reused by subclasses.
  • Maintainability: Changes to common logic in the superclass automatically propagate to subclasses.
  • Polymorphism: Enables treating objects of different subclasses uniformly through a superclass reference.
  • Extensibility: New subclasses can be added easily without modifying existing code.

The `super` Keyword:

  • Used to refer to the immediate parent class object.
  • Can be used to:
    • Call the parent class's constructor (`super(...)`). Must be the first statement in the subclass constructor.
    • Access parent class members (fields or methods) that are hidden or overridden by the subclass (`super.field` or `super.method()`).

Example: Animal & Dog Hierarchy

Animal.java
class Animal { String name; int age; public Animal(String name, int age) { this.name = name; this.age = age; System.out.println("Animal constructor: " + name); } public void eat() { System.out.println(name + " is eating."); } public void sleep() { System.out.println(name + " is sleeping."); } } class Dog extends Animal { String breed; public Dog(String name, int age, String breed) { super(name, age); // Call Animal's constructor this.breed = breed; System.out.println("Dog constructor: " + name + ", breed " + breed); } public void bark() { System.out.println(name + " is barking!"); } } public class Main { public static void main(String[] args) { Dog myDog = new Dog("Buddy", 3, "Golden Retriever"); myDog.eat(); // Inherited from Animal myDog.sleep(); // Inherited from Animal myDog.bark(); // Specific to Dog } }

**Trainer Note:** "Notice how `Dog` automatically gets `eat()` and `sleep()` from `Animal`. This saves us from writing that code again. Also, see how `super()` is used to call the `Animal` constructor first."

Polymorphism in Action

Method Overloading & Overriding

Method Overloading: Same Name, Different Parameters

  • Allows a class to have multiple methods with the same name but different parameter lists (different number, type, or order of parameters).
  • Occurs within the same class.
  • Determined at compile time (static polymorphism).
  • Return type can be different, but not the sole distinguishing factor. If parameters are the same, return type *must* be the same.
  • Example: `add(int a, int b)` vs. `add(double a, double b)` vs. `add(int a, int b, int c)`.

Method Overriding: Same Signature, Different Implementation

  • Allows a subclass to provide a specific implementation for a method that is already defined in its superclass.
  • Occurs in an inheritance hierarchy (subclass overriding superclass method).
  • Method signature (name, number, and type of parameters) and return type must be the same (or a covariant return type).
  • Determined at runtime (dynamic polymorphism).
  • Use the `@Override` annotation for clarity and compile-time checks.
💡
Overloading provides flexibility for a single method name to handle different data inputs. Overriding allows specialized behavior in subclasses while maintaining a common interface.

Example: Overloading & Overriding

ShapeCalculator.java
class Shape { public double area() { return 0.0; // Default implementation } public void draw() { System.out.println("Drawing a generic shape."); } } class Circle extends Shape { double radius; public Circle(double radius) { this.radius = radius; } @Override // Overriding area() from Shape public double area() { return Math.PI * radius * radius; } // Overloaded draw methods public void draw() { // Overriding draw() from Shape System.out.println("Drawing a circle."); } public void draw(String color) { // Overloaded draw method System.out.println("Drawing a " + color + " circle."); } } public class Main { public static void main(String[] args) { Circle c = new Circle(5); System.out.println("Circle Area: " + c.area()); // Calls overridden area() c.draw(); // Calls overridden draw() c.draw("Red"); // Calls overloaded draw(String) } }

Advanced Abstraction

Abstract Classes & Interfaces

Abstract Classes: Partial Blueprints

  • A class that is declared with the `abstract` keyword.
  • Cannot be instantiated directly (you cannot create objects of an abstract class).
  • Can have `abstract` methods (methods without a body) AND concrete methods (methods with a body).
  • Subclasses must implement all abstract methods of the abstract parent, or they must also be declared abstract.
  • Used when you want to provide a common base class for subclasses, but some behavior needs to be defined by each subclass individually. Represents an "is-a" relationship with shared state and partially implemented behavior.
  • Example: `abstract class Shape { abstract double area(); String color; }`

Interfaces: Pure Abstraction (Contracts)

  • A blueprint of a class that can contain only abstract methods (pre-Java 8), `default` methods, `static` methods, and constant variables.
  • Declared with the `interface` keyword.
  • Cannot be instantiated directly.
  • A class `implements` an interface, agreeing to provide implementations for all its abstract methods.
  • Represents a "can-do" or "has-a" capability. A class can implement multiple interfaces (solving Java's single inheritance limitation for behavior).
  • Example: `interface Drawable { void draw(); }`

Comparison: Abstract Class vs. Interface

Feature Abstract Class Interface
Instantiation No No
Abstract Methods Yes Yes
Concrete Methods Yes Yes (default/static in Java 8+)
Fields Any access modifier public static final (implicitly)
Inheritance One class can extend one abstract class One class can implement multiple interfaces
💡
Abstract classes are for *specializing* a type (e.g., a "Vehicle" base for "Car" and "Bike"). Interfaces are for *defining capabilities* (e.g., "Driveable" for anything that can be driven).

**Trainer Note:** "These are fundamental for designing flexible and extensible architectures. Understanding when to use which is a sign of a seasoned Java developer."

Dynamic Behavior

Runtime Polymorphism: Dynamic Method Dispatch

What is Runtime Polymorphism?

  • Literally means "many forms" at runtime.
  • In Java, it's primarily achieved through method overriding.
  • It allows a program to decide which method implementation to call at runtime, rather than at compile time.
  • This is also known as Dynamic Method Dispatch.
  • A superclass reference variable can refer to objects of its subclass. When an overridden method is called using this reference, the version of the method in the actual object type (subclass) is executed.
  • Example: An `Animal` reference variable holding a `Dog` object will call `Dog`'s `makeSound()` method if `Dog` overrides it.

Benefits:

  • Flexibility & Extensibility: Write code that can work with objects of various types related by inheritance. Easily add new subclasses without modifying existing code.
  • Decoupling: Reduces the coupling between client code and specific implementations, allowing for more modular and maintainable systems.
  • Achieves "Plug-and-Play" Architecture: New components (subclasses) can "plug in" to an existing system by adhering to the superclass's contract.
💡
Runtime Polymorphism is a cornerstone of modern OOP, enabling highly flexible and adaptable software designs. It's why interfaces and abstract classes are so powerful.

Example: Runtime Polymorphism

PolymorphismDemo.java
class Animal { public void makeSound() { System.out.println("Animal makes a generic sound."); } } class Dog extends Animal { @Override public void makeSound() { System.out.println("Woof! Woof!"); } } class Cat extends Animal { @Override public void makeSound() { System.out.println("Meow."); } } public class Main { public static void main(String[] args) { Animal myAnimal; // Declare a reference of type Animal myAnimal = new Dog(); // Assign a Dog object myAnimal.makeSound(); // Calls Dog's makeSound() at runtime myAnimal = new Cat(); // Assign a Cat object myAnimal.makeSound(); // Calls Cat's makeSound() at runtime Animal[] animals = new Animal[3]; animals[0] = new Dog(); animals[1] = new Cat(); animals[2] = new Animal(); for (Animal a : animals) { a.makeSound(); // Each calls its own specific implementation } } }

OOP Best Practices & Clean Code Principles

1. DRY (Don't Repeat Yourself)

  • Avoid code duplication. Use inheritance, composition, and helper methods.
  • Example: Constructor chaining with `this()` and `super()` helps reduce repetitive initialization code.

2. SOLID Principles

A set of five design principles intended to make software designs more understandable, flexible, and maintainable.

  • Single Responsibility Principle (SRP)
  • Open/Closed Principle (OCP)
  • Liskov Substitution Principle (LSP)
  • Interface Segregation Principle (ISP)
  • Dependency Inversion Principle (DIP)

3. Encapsulate Everything

  • Keep fields `private` and provide access through `public` getters and setters.
  • This protects internal state and allows for validation.

4. Favor Composition Over Inheritance

  • Inheritance creates tight coupling. Composition (using objects as fields) offers more flexibility.
  • "Has-a" relationships (`Car has an Engine`) are often better as composition than "is-a" (`Car is a Vehicle`).

5. Use Interfaces for Contracts

  • Define behavior contracts with interfaces. This allows for flexible implementations and polymorphic code.
  • Promotes loose coupling between modules.

6. Write Readable, Maintainable Code

  • Meaningful variable/method names.
  • Consistent formatting (use an IDE's formatter).
  • Clear comments for complex logic (but aim for self-documenting code).
  • Break down large methods into smaller, focused ones.
💡
Adhering to these principles leads to more robust, scalable, and easier-to-debug applications. They're not just theoretical; they're essential for production-grade code.
