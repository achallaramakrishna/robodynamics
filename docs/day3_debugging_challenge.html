Debug Challenge Lab

Each challenge describes a subtle bug. Give learners 7-10 minutes, then walk through the fix together and highlight best practices.

Challenge 1

Handle a lazy cache peek

Users report occasional NullPointerExceptions when multiple threads load cache data.

public Integer getScore(String learnerId) {
  return learnerScores.get(learnerId).intValue();
}
  • Gap: no null-check before unboxing and `get` can return null for new entries.
  • Fix: use `Objects.requireNonNullElse` or default to `0`, then unbox safely.
  • Explain: unboxing null throws NPE; best to return a default, log missing keys, and avoid global state mutation.
Challenge 2

Misordered multi-catch

Compilation fails even though each exception is handled.

try {
  Files.readAllLines(path);
} catch (IOException e) {
  log(e);
} catch (FileNotFoundException fnf) {
  log(fnf);
}
  • Java checks catch blocks top to bottom; subclasses must appear before supertypes.
  • Swap order or use multi-catch to keep `FileNotFoundException` reachable.
  • Explain: FileNotFoundException is a subtype of IOException, so compiler sees unreachable block.
Challenge 3

Unchecked cast in a generic factory

Generic factory returns unchecked warnings and fails when retrieving typed data.

public <T> T loadValue(Class<T> type) {
  Map<String, Object> storage = loadStorage();
  return (T) storage.get(type.getSimpleName());
}
  • Problem: raw map entries get cast without verification, so a ClassCastException may surface.
  • Suggestion: store values using `Map, Object>` and even check `type.isInstance(value)` before casting.
  • Explain: generics are compile-time; retaining `Class` lets you guard at runtime before the cast.
