Exception Handling

Exception Handling: The "Why" & Basic Flow

What is an Exception?

An **exception** is an event that disrupts the normal flow of a program's instructions. It's an object that encapsulates information about an abnormal event that occurred during execution.

  • **Error vs. Exception:**
    • `Error`: Represents serious problems that most applications should not try to catch (e.g., `OutOfMemoryError`, `StackOverflowError`).
    • `Exception`: Represents conditions that a reasonable application might want to catch (e.g., `IOException`, `NullPointerException`).

**Why handle them?**

  • **Graceful Degradation:** Prevent program crashes.
  • **User Experience:** Provide meaningful error messages.
  • **Recovery:** Allow the program to attempt recovery from an error.
  • **Debugging:** Centralize error reporting and logging.

Basic Exception Handling (`try-catch-finally`)


// Basic structure
try {
    // Code that might throw an exception
    int result = 10 / 0; // This will throw an ArithmeticException
    System.out.println("Result: " + result); // This line won't be reached
} catch (ArithmeticException e) {
    // Code to handle the specific exception
    System.err.println("Caught an ArithmeticException: " + e.getMessage());
} catch (Exception e) { // Generic catch for any other unexpected exception
    System.err.println("Caught a generic Exception: " + e.getMessage());
} finally {
    // Code that always executes, regardless of whether an exception occurred or not
    System.out.println("Finally block executed. Cleanup tasks go here.");
}
System.out.println("Program continues after the try-catch-finally block.");

Exception Handling

Checked vs. Unchecked: The Compiler's Role

Checked Exceptions (Compile-time Exceptions)

  • **Definition:** Exceptions that must be caught or declared in the method signature using the `throws` keyword. The compiler enforces this.
  • **Hierarchy:** Subclasses of `java.lang.Exception` (but not `RuntimeException`).
  • **Examples:** `IOException`, `SQLException`, `ClassNotFoundException`.
  • **Use Case:** Typically represent problems outside the immediate control of the program that can reasonably be recovered from (e.g., file not found, network issues).

**Rule:** If a method calls another method that declares a checked exception, the calling method must either `catch` it or `declare` it (pass it up the call stack).

Unchecked Exceptions (Runtime Exceptions)

  • **Definition:** Exceptions that do NOT need to be caught or declared. The compiler does not enforce their handling.
  • **Hierarchy:** Subclasses of `java.lang.RuntimeException` (which is a subclass of `Exception`).
  • **Examples:** `NullPointerException`, `ArithmeticException`, `ArrayIndexOutOfBoundsException`, `IllegalArgumentException`.
  • **Use Case:** Typically represent programming errors (bugs) that should ideally be fixed rather than caught (e.g., trying to access an array out of bounds, dereferencing a null object).

**Rule:** It's generally a bad practice to catch `RuntimeException`s unless you have a specific recovery strategy. They indicate a flaw in logic.

Exception Handling

Declaring & Propagating: `throws` Keyword

Handling Checked Exceptions: `try-catch` vs. `throws`

When a method might throw a checked exception, you have two primary choices:

  • **`try-catch`:** Handle the exception immediately within the method where it might occur. This is suitable if the method itself knows how to recover or respond to the exception.
  • **`throws` keyword:** Declare that the method *might* throw the exception, delegating the responsibility of handling it to the calling method. This is useful when the current method cannot meaningfully handle the exception, and a higher-level caller is better equipped.

Coding Scenario: `throws` in Action


import java.io.FileReader;
import java.io.IOException;

public class ThrowsKeywordDemo {

    // Method 1: Declares (throws) an IOException, passing responsibility to its caller
    public void readFile(String filePath) throws IOException {
        System.out.println("Attempting to read file: " + filePath);
        FileReader reader = null;
        try {
            reader = new FileReader(filePath);
            // Simulate reading the file
            int charCode;
            while ((charCode = reader.read())!= -1) {
                System.out.print((char) charCode);
            }
            System.out.println("\nFile read successfully.");
        } finally {
            if (reader!= null) {
                reader.close(); // FileReader.close() can also throw IOException!
            }
        }
    }

    // Method 2: Catches the IOException thrown by readFile
    public void processFile(String fileName) {
        try {
            readFile(fileName);
        } catch (IOException e) {
            System.err.println("Error processing file '" + fileName + "': " + e.getMessage());
            // Log the error, notify user, or try an alternative
        }
    }

    public static void main(String[] args) {
        ThrowsKeywordDemo demo = new ThrowsKeywordDemo();
        // This will succeed if "my_file.txt" exists and is readable
        demo.processFile("my_file.txt");

        // This will cause an IOException (file not found)
        demo.processFile("non_existent_file.txt");
    }
}

Exception Handling

Multi-Catch & `finally`: Refined Handling

Catching Multiple Exceptions (Java 7+)

Before Java 7, if you wanted to handle multiple distinct exception types in the same way, you had to write separate `catch` blocks for each. With multi-catch, you can catch several exceptions in a single `catch` block if they are unrelated or share a common handling logic.

  • **Syntax:** `catch (ExceptionType1 | ExceptionType2 |... e)`
  • **Benefit:** Reduces code duplication and improves readability.
  • **Restriction:** The exception types in a multi-catch block cannot be subclasses or superclasses of each other (e.g., `catch (IOException | FileNotFoundException e)` is invalid because `FileNotFoundException` is a subclass of `IOException`).

The `finally` Block: Guaranteed Execution

The `finally` block is always executed regardless of whether an exception occurs in the `try` block or not, and regardless of whether it's caught. It's primarily used for cleanup operations.

  • **Purpose:** Close resources (files, network connections, database connections), release locks, etc.
  • **Guaranteed?** Almost always. Exceptions: `System.exit()` call, JVM crash, infinite loop in `try` or `catch`.

Coding Scenario: Multi-Catch & `finally`


import java.io.FileInputStream;
import java.io.IOException;
import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class MultiCatchFinallyDemo {

    public static void connectAndLoad(String dbUrl, String filePath) {
        Connection conn = null;
        FileInputStream fis = null;

        try {
            // Simulate database connection
            conn = DriverManager.getConnection(dbUrl); // Might throw SQLException
            System.out.println("Database connected.");

            // Simulate file loading
            fis = new FileInputStream(filePath); // Might throw FileNotFoundException (an IOException)
            int data = fis.read(); // Might throw IOException
            System.out.println("File data read: " + data);

        } catch (SQLException | IOException e) { // Multi-catch for related but distinct exceptions
            System.err.println("An error occurred during DB or File operation: " + e.getMessage());
            // In real app: log the exception, show user-friendly message
        } finally {
            System.out.println("Executing finally block for cleanup...");
            // Close resources in reverse order of opening
            try {
                if (fis!= null) fis.close();
                System.out.println("FileInputStream closed.");
            } catch (IOException e) {
                System.err.println("Error closing file: " + e.getMessage());
            }
            try {
                if (conn!= null) conn.close();
                System.out.println("Database connection closed.");
            } catch (SQLException e) {
                System.err.println("Error closing database connection: " + e.getMessage());
            }
        }
        System.out.println("Method finished.");
    }

    public static void main(String[] args) {
        System.out.println("--- Scenario 1: All operations succeed ---");
        // These might fail in a real setup, but for demo, assume success
        connectAndLoad("jdbc:mock:db", "existing_file.txt");

        System.out.println("\n--- Scenario 2: File Not Found ---");
        connectAndLoad("jdbc:mock:db", "non_existent_file.txt");
    }
}

Exception Handling

`try-with-resources`: Automatic Resource Management

The Problem with Manual Resource Closing

Manually closing resources in `finally` blocks (as seen in the previous slide) is verbose and error-prone, especially when multiple resources are involved. If one `close()` call throws an exception, others might not be executed.

  • **Boilerplate Code:** Repetitive `if (resource!= null) { resource.close(); }` blocks.
  • **Suppressed Exceptions:** If an exception is thrown in the `try` block, and another is thrown in the `finally` block during cleanup, the original exception can be "lost" (suppressed).

`try-with-resources` (TWR) - Java 7+

TWR automatically closes resources that implement the `java.lang.AutoCloseable` interface (or `java.io.Closeable`, which extends `AutoCloseable`).

  • **Syntax:** Resources are declared within the `try` parentheses.
  • **Automatic Closing:** Resources are closed automatically at the end of the `try` block, regardless of how the block exits (normally or via an exception).
  • **Exception Handling:** If an exception occurs in the `try` block and another during `close()`, the exception from `close()` is suppressed, and the original exception is re-thrown. The suppressed exception can be retrieved via `Throwable.getSuppressed()`.

Coding Scenario: `try-with-resources` for `FileReader`


import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class TryWithResourcesDemo {

    public static void readFileTWR(String filePath) {
        // Resources declared here are automatically closed
        try (FileReader fr = new FileReader(filePath);
             BufferedReader br = new BufferedReader(fr)) { // Multiple resources allowed

            System.out.println("Reading from file: " + filePath);
            String line;
            while ((line = br.readLine())!= null) {
                System.out.println(" " + line);
            }
            System.out.println("File read successfully.");
        } catch (IOException e) { // Catch block is still needed for declared exceptions
            System.err.println("Error reading file: " + e.getMessage());
            // Exception in fr.close() or br.close() would be suppressed and added to 'e'
            for (Throwable suppressed : e.getSuppressed()) {
                System.err.println(" Suppressed Exception: " + suppressed.getMessage());
            }
        }
        System.out.println("Method finished.");
    }

    public static void main(String[] args) {
        System.out.println("--- Scenario 1: Existing File ---");
        // Create an "example.txt" file for this to work
        readFileTWR("example.txt");

        System.out.println("\n--- Scenario 2: Non-Existing File ---");
        readFileTWR("non_existent_file.txt");
    }
}

Exception Handling

Custom Exceptions: Tailoring Your Errors

Why Create Custom Exceptions?

  • Business Logic Errors: Standard Java exceptions don't always accurately represent domain-specific problems (e.g., "Insufficient Funds" is not an `IllegalArgumentException`).
  • Improved Readability & Clarity: Custom exceptions make the code's intent clearer to other developers.
  • Fine-grained Handling: Allows client code to catch and handle specific business errors more precisely.

Rule of thumb: If a standard `RuntimeException` (like `IllegalArgumentException` or `IllegalStateException`) doesn't perfectly describe the error, consider a custom exception.

How to Create & Use Custom Exceptions

Your custom exception class should extend either `java.lang.Exception` (for a **checked** exception) or `java.lang.RuntimeException` (for an **unchecked** exception).

  • Extend `Exception`: Use for conditions client code *must* explicitly handle (e.g., file not found from an external source, network issues). Signals a *recoverable* problem.
  • Extend `RuntimeException`: Use for programming errors or situations client code *should* ideally prevent. Signals an *unrecoverable* problem that often indicates a bug.

// 1. Define your Custom Exception (Checked)
class InsufficientFundsException extends Exception {
    private double currentBalance;
    private double withdrawalAmount;

    public InsufficientFundsException(String message, double currentBalance, double withdrawalAmount) {
        super(message);
        this.currentBalance = currentBalance;
        this.withdrawalAmount = withdrawalAmount;
    }

    public double getCurrentBalance() { return currentBalance; }
    public double getWithdrawalAmount() { return withdrawalAmount; }
}

// 2. Use it in a Service Class
class BankAccount {
    private double balance;

    public BankAccount(double initialBalance) {
        this.balance = initialBalance;
    }

    public void withdraw(double amount) throws InsufficientFundsException { // Declare it!
        if (amount <= 0) {
            throw new IllegalArgumentException("Withdrawal amount must be positive.");
        }
        if (balance < amount) {
            throw new InsufficientFundsException(
                "Attempted withdrawal of " + amount + " but balance is only " + balance,
                balance, amount
            );
        }
        balance -= amount;
        System.out.println("Withdrew " + amount + ". New balance: " + balance);
    }
}

// 3. Handle it in Client Code
public class CustomExceptionDemo {
    public static void main(String[] args) {
        BankAccount account = new BankAccount(500);

        try {
            account.withdraw(200); // OK
            account.withdraw(400); // Insufficient funds!
            account.withdraw(-50); // IllegalArgumentException (Unchecked)
        } catch (InsufficientFundsException e) {
            System.err.println("Transaction Failed: " + e.getMessage());
            System.err.println("Current Balance: " + e.getCurrentBalance() + ", Attempted: " + e.getWithdrawalAmount());
        } catch (IllegalArgumentException e) { // Catches the unchecked exception
            System.err.println("Input Error: " + e.getMessage());
        } catch (Exception e) { // Generic catch for any other unexpected issues
            System.err.println("An unexpected error occurred: " + e.getMessage());
        }
    }
}

Exception Handling

`if` vs. `try-catch`: Deciding Your Strategy

The Core Principle: Predictability and Recoverability

  • Use `if` for predictable, preventable conditions:
    • When you can check a condition *before* an operation, and the failure is a normal, expected deviation in flow (e.g., input validation).
    • This prevents an exception from being thrown in the first place.
    • Leads to cleaner code and often better performance (exceptions are expensive).
  • Use `try-catch` for unpredictable, exceptional conditions:
    • When an operation might fail due to circumstances outside your control or that are genuinely "exceptional" (e.g., network down, file missing, external service error).
    • When the failure is not a normal part of your application's logic flow.
    • When you need to *recover* from an error or log it appropriately.

Trainer Note: This distinction is crucial for writing efficient, readable, and robust code. Don't use exceptions for flow control!

Scenario: Parsing User Input (Comparing Approaches)


public class IfVsTryCatch {

    // Approach 1: Using 'if' for validation (PREFERABLE for expected input issues)
    public static void processInputWithIf(String input) {
        System.out.println("\n--- Processing with IF ---");
        if (input == null || input.trim().isEmpty()) {
            System.out.println("Error: Input cannot be empty.");
            return;
        }
        if (!input.matches("\\d+")) { // Check if it's purely digits
            System.out.println("Error: Input '" + input + "' is not a valid number (contains non-digits).");
            return;
        }

        int number = Integer.parseInt(input); // Now it's safer to parse
        System.out.println("Successfully parsed (if-check): " + number);
    }

    // Approach 2: Relying on 'try-catch' (Use for truly unexpected external input failures)
    public static void processInputWithTryCatch(String input) {
        System.out.println("\n--- Processing with TRY-CATCH ---");
        try {
            int number = Integer.parseInt(input);
            System.out.println("Successfully parsed (try-catch): " + number);
        } catch (NumberFormatException e) {
            System.out.println("Error: Input '" + input + "' is not a valid number (NumberFormatException).");
            // Log.error("Failed to parse input", e); // In a real project, always log!
        } catch (Exception e) {
            System.out.println("An unexpected error occurred: " + e.getMessage());
        }
    }

    public static void main(String[] args) {
        processInputWithIf("123");
        processInputWithIf("");
        processInputWithIf("abc");

        processInputWithTryCatch("456");
        processInputWithTryCatch(""); // This will parse as empty string, then fail parseInt
        processInputWithTryCatch("xyz");
    }
}

Exception Handling

Exception Handling: In Project Architecture

Beyond Basic `try-catch`: Production-Ready Practices

  • Logging, Not Printing: Never use `e.printStackTrace()` or `System.err.println()` in production code. Use a proper logging framework (e.g., SLF4J with Logback/Log4j2).
  • Don't Swallow Exceptions: If you `catch` an exception, *do something* with it – log it, re-throw it (possibly wrapped), or present a user-friendly error. An empty `catch` block is a major anti-pattern.
  • Wrap & Re-throw: Often, you catch a low-level exception (e.g., `SQLException`) and wrap it in a custom, higher-level, business-specific exception (e.g., `ProductPersistenceException`) to provide more context.
  • Global Exception Handlers: In frameworks like Spring Boot, you define centralized mechanisms to catch unhandled exceptions at a high level and return consistent error responses (e.g., JSON error objects for APIs).

Trainer Note: Good exception handling is a cornerstone of maintainable and scalable enterprise applications.

Example: Logging vs. `printStackTrace()`

Using a logger (like SLF4J/Logback) provides powerful features like filtering, different log levels (DEBUG, INFO, WARN, ERROR), and output to various destinations (console, file, network).


// Example: Using SLF4J Logger (replace with your actual logger setup)
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class ProjectExceptionHandling {

    private static final Logger logger = LoggerFactory.getLogger(ProjectExceptionHandling.class);

    public void fetchData(String resourceId) {
        try {
            // Simulate an operation that might fail
            if (resourceId.equals("invalid")) {
                throw new IOException("Resource not found or inaccessible: " + resourceId);
            }
            logger.info("Data fetched successfully for resource: {}", resourceId);
        } catch (IOException e) {
            // Instead of e.printStackTrace();
            logger.error("Failed to fetch data for resource: {}", resourceId, e); // Log message, resourceId, AND the exception stack trace
            // Potentially re-throw a custom exception
            throw new RuntimeException("Data access failed due_to_io_error", e);
        }
    }

    public static void main(String[] args) {
        ProjectExceptionHandling handler = new ProjectExceptionHandling();
        handler.fetchData("valid_resource");
        try {
            handler.fetchData("invalid");
        } catch (RuntimeException e) {
            System.err.println("Caught wrapped exception in main: " + e.getMessage());
        }
    }
}

Collections Framework

Collections Hierarchy & List Implementations

The Java Collections Framework: Your Data Toolbox

  • A unified architecture for representing and manipulating collections of objects.
  • Provides interfaces (e.g., `List`, `Set`, `Map`) and their implementations (e.g., `ArrayList`, `HashSet`, `HashMap`).
  • **Goal:** To enable interoperability, reduce programming effort, and increase performance.

Key Interfaces: `Collection` (root), `List`, `Set`, `Queue`, and `Map` (separate hierarchy).

`List` Interface: Ordered, Indexed, Allows Duplicates

Represents an ordered collection (sequence) of elements. Elements can be accessed by their integer index.

`ArrayList`: Fast Random Access (like a dynamic array)

  • Internal: Resizable array.
  • Best For: Frequent random access (`get(index)` is O(1)).
  • Performance:
    • `add(element)` at end: Amortized O(1).
    • `get(index)`: O(1).
    • `add(index, element)` in middle: O(n) (requires shifting elements).
    • `remove(index)` in middle: O(n).

`LinkedList`: Fast Inserts/Deletes (like a doubly-linked list)

  • Internal: Doubly-linked list.
  • Best For: Frequent insertions and deletions in the middle.
  • Performance:
    • `add(element)` at end: O(1).
    • `get(index)`: O(n) (requires traversal).
    • `add(index, element)` in middle: O(1) (after O(n) traversal to find index).
    • `remove(index)` in middle: O(1) (after O(n) traversal).

Coding Scenario: Choosing the Right `List`


import java.util.ArrayList;
import java.util.LinkedList;
import java.util.List;

public class ListChoiceDemo {
    public static void main(String[] args) {
        // Scenario 1: You need to frequently access elements by index (e.g., display by row number)
        List<String> shoppingList = new ArrayList<>();
        shoppingList.add("Milk");
        shoppingList.add("Bread");
        shoppingList.add("Eggs");
        System.out.println("Item 2 (0-indexed): " + shoppingList.get(1)); // Fast access

        // Scenario 2: You frequently add/remove items from the middle (e.g., managing a playlist)
        List<String> playlist = new LinkedList<>();
        playlist.add("Song A");
        playlist.add("Song C");
        playlist.add(1, "Song B"); // Efficient insertion in middle
        System.out.println("Playlist: " + playlist);
        playlist.remove("Song A"); // Efficient removal
        System.out.println("Updated Playlist: " + playlist);
    }
}

Collections Framework

`List` Interface: Key Methods & Utilities

Beyond `add()` and `get()`: Writing Concise Code

Knowing common `List` methods helps write efficient and readable code, often turning multiple lines of manual iteration into a single method call.

  • `indexOf(Object o)`: Returns the index of the first occurrence of the specified element, or -1 if this list does not contain the element.
  • `lastIndexOf(Object o)`: Returns the index of the last occurrence.
  • `subList(int fromIndex, int toIndex)`: Returns a `view` of the portion of this list between the specified `fromIndex`, inclusive, and `toIndex`, exclusive. Changes to the sublist are reflected in the original list and vice versa.
  • `replaceAll(UnaryOperator operator)`: Replaces each element of this list with the result of applying the operator to that element. (Java 8+)
  • `sort(Comparator c)`: Sorts this list according to the order induced by the specified `Comparator`. (Java 8+)

Trainer Note: Leveraging these built-in methods is crucial for writing "3 lines of code" instead of "20 lines of code" in projects.

Practical Examples with `ArrayList`


import java.util.ArrayList;
import java.util.Collections;
import java.util.List;
import java.util.Comparator;

public class ListMethodsDemo {
    public static void main(String[] args) {
        List<String> names = new ArrayList<>();
        names.add("Alice");
        names.add("Bob");
        names.add("Charlie");
        names.add("Alice"); // Duplicate

        System.out.println("Original List: " + names);

        // 1. indexOf and lastIndexOf
        System.out.println("First 'Alice' at: " + names.indexOf("Alice"));
        System.out.println("Last 'Alice' at: " + names.lastIndexOf("Alice"));
        System.out.println("'David' at: " + names.indexOf("David")); // -1

        // 2. subList - creating a view
        List<String> sub = names.subList(1, 3); // Elements at index 1 and 2
        System.out.println("Sublist (index 1 to 2): " + sub);
        sub.add("Eve"); // Adds to original 'names' list too!
        System.out.println("Original List after sublist modification: " + names);

        // 3. replaceAll (Java 8+)
        List<String> fruits = new ArrayList<>(List.of("apple", "banana", "cherry"));
        fruits.replaceAll(String::toUpperCase); // Convert all to uppercase
        System.out.println("Fruits after replaceAll: " + fruits);

        // 4. sort (Java 8+) - using default String comparator
        names.sort(Comparator.naturalOrder()); // Sorts alphabetically
        System.out.println("Names after sort: " + names);

        // Another sort example: sort by length (Java 8+)
        fruits.sort(Comparator.comparingInt(String::length));
        System.out.println("Fruits sorted by length: " + fruits);
    }
}

Collections Framework

`Set` Interface: Unique Elements Only

The `Set` Contract: No Duplicate Elements

A `Set` is a collection that cannot contain duplicate elements. It models the mathematical set abstraction. There is no specific order of elements (though some implementations guarantee iteration order).

Choosing the right `Set` implementation depends on whether you need ordering and the performance characteristics you prioritize.

`HashSet`: Fast Access, No Order Guaranteed

  • Internal: Backed by a `HashMap`.
  • Characteristics: Offers constant time performance (O(1)) for basic operations (add, remove, contains), assuming the hash function disperses elements properly. Does not guarantee order.
  • Key Requirement: Elements must correctly implement `hashCode()` and `equals()` methods for uniqueness.
  • Best For: High-performance storage of unique elements when iteration order is not important.

`LinkedHashSet`: Insertion Order Maintained

  • Internal: Backed by a `LinkedHashMap`.
  • Characteristics: Maintains the insertion order of elements (they iterate in the order they were inserted). Provides near `HashSet` performance.
  • Best For: When you need unique elements but also need to preserve the order in which they were added.

`TreeSet`: Sorted Order Maintained

  • Internal: Backed by a `TreeMap`.
  • Characteristics: Stores elements in a sorted (natural or custom) order. Operations typically take O(log n) time. Does not allow `null` keys.
  • Key Requirement: Elements must implement `Comparable` or a `Comparator` must be provided at construction time.
  • Best For: When you need unique elements and they must always be kept in sorted order.

`Set` Interface: Practical Scenarios

Leveraging `Set` for Common Problems

  • Removing Duplicates: The most common use case. Copying elements from a `List` to a `Set` (and back if needed) is the most efficient way to get unique elements.

Collections Framework

`Map` Interface: Key-Value Powerhouses

The `Map` Contract: Unique Keys Map to Values

A `Map` is an object that maps keys to values. A `Map` cannot contain duplicate keys; each key can map to at most one value. It's not a `Collection` but an essential part of the Java Collections Framework.

Choosing the right `Map` depends on performance, whether you need key ordering, and how you handle nulls.

`HashMap`: Fast, Unordered, Allows Nulls

  • Internal: Backed by a hash table (array of linked lists/trees).
  • Characteristics: Provides constant time performance (O(1)) for basic operations (`get`, `put`, `contains`) in average case. No guaranteed order. Allows one `null` key and multiple `null` values.
  • Key Requirement: Keys must correctly implement `hashCode()` and `equals()` for proper functioning.
  • Best For: High-performance lookups when order isn't a concern. Most common `Map` implementation.

`LinkedHashMap`: Insertion Order Maintained, Fast

  • Internal: Hash table with a doubly-linked list running through its entries.
  • Characteristics: Maintains the insertion order of key-value pairs (they iterate in the order they were inserted). Offers near `HashMap` performance.
  • Best For: When you need the fast lookups of `HashMap` but also need to preserve the order in which entries were added.

`TreeMap`: Sorted by Keys, O(log n)

  • Internal: Backed by a Red-Black tree.
  • Characteristics: Stores entries in a sorted order based on the keys (natural or custom). Operations typically take O(log n) time. Does not allow `null` keys.
  • Key Requirement: Keys must implement `Comparable` or a `Comparator` must be provided.
  • Best For: When you need keys to be automatically sorted.

`Map` Interface: Leveraging Advanced Methods

Beyond Basic `put()` and `get()`: Writing Concise Logic (Java 8+)

Modern Java introduced several powerful default methods to the `Map` interface, making common patterns more concise and efficient. These methods often replace verbose `if/else` checks for existence.

  • `getOrDefault(Object key, V defaultValue)`: Returns the value for the specified key, or `defaultValue` if the key is not found. Avoids `NullPointerException` if key doesn't exist.
  • `putIfAbsent(K key, V value)`: Inserts the specified value with the specified key only if the key is not already associated with a value. Returns the previous value associated with the key, or `null` if there was no mapping.
  • `computeIfAbsent(K key, Function mappingFunction)`: If the specified key is not already associated with a value (or is mapped to `null`), attempts to compute its value using the given mapping function and enters it into this map.
  • `computeIfPresent(K key, BiFunction remappingFunction)`: If the value for the specified key is present and non-null, attempts to compute a new mapping given the key and its current mapped value.
  • `merge(K key, V value, BiFunction remappingFunction)`: If the specified key is not already associated with a value or is associated with `null`, associates it with the given non-null `value`. Otherwise, it attempts to re-map the value using the given remapping function.

Trainer Note: Mastering these methods can drastically reduce boilerplate code, especially in scenarios like frequency counting or grouping data.

        String[] words = text.toLowerCase().replaceAll("[^a-z ]", "").split(" "); // Clean and split
        Map<String, Integer> wordFrequencies = new HashMap<>();

        // --- Traditional way (pre-Java 8) ---
        // for (String word : words) {
        // if (!word.isEmpty()) {
        // Integer count = wordFrequencies.get(word);
        // if (count == null) {
        // wordFrequencies.put(word, 1);

        }

        // --- Using getOrDefault ---

`Queue` Interface: Processing Pipelines (FIFO)

The `Queue` Contract: First-In, First-Out (FIFO)

A `Queue` is a collection designed for holding elements prior to processing. Besides basic `Collection` operations, queues provide additional insertion, extraction, and inspection operations.

  • **FIFO (First-In, First-Out):** Elements are typically processed in the order they were added.
  • Common Implementations:
    • `LinkedList`: Can function as a `Queue` and `Deque`.
    • `ArrayDeque`: A more efficient double-ended queue, often preferred over `LinkedList` for queue behavior.
    • `PriorityQueue`: Elements are ordered according to their natural ordering, or by a `Comparator`. Not FIFO in the traditional sense, but by priority.

Key `Queue` Operations (Two Forms)

Queues typically offer two sets of methods for each operation:

Operation Throws Exception Returns Special Value
Insert `add(e)` `offer(e)`
Remove `remove()` `poll()`
Examine `element()` `peek()`

Trainer Note: Use `offer`, `poll`, `peek` (return special value) to avoid exceptions for normal flow; use `add`, `remove`, `element` (throw exception) when an empty queue is truly an error condition.

Coding Scenario: Simulating a Simple Ticket Queue


import java.util.ArrayDeque;
import java.util.Queue;

public class TicketQueueDemo {
    public static void main(String[] args) {
        Queue<String> ticketQueue = new ArrayDeque<>();

        // Add people to the queue (offer is generally safer than add)
        ticketQueue.offer("Alice");
        ticketQueue.offer("Bob");
        ticketQueue.offer("Charlie");
        System.out.println("Initial Queue: " + ticketQueue); // [Alice, Bob, Charlie]

        // Peek at the next person without removing
        String nextPerson = ticketQueue.peek();
        System.out.println("Next person in queue (peek): " + nextPerson); // Alice
        System.out.println("Queue after peek: " + ticketQueue); // [Alice, Bob, Charlie]

        // Process people from the queue (poll is generally safer than remove)
        String processedPerson1 = ticketQueue.poll();
        System.out.println("Processed: " + processedPerson1); // Alice
        System.out.println("Queue after first poll: " + ticketQueue); // [Bob, Charlie]

        String processedPerson2 = ticketQueue.poll();
        System.out.println("Processed: " + processedPerson2); // Bob
        System.out.println("Queue after second poll: " + ticketQueue); // [Charlie]

        // Check if queue is empty
        System.out.println("Is queue empty? " + ticketQueue.isEmpty());

        // Process last person
        String processedPerson3 = ticketQueue.poll();
        System.out.println("Processed: " + processedPerson3); // Charlie
        System.out.println("Queue after third poll: " + ticketQueue); // []

        // What happens when polling an empty queue?
        String processedPerson4 = ticketQueue.poll();
        System.out.println("Processed from empty queue: " + processedPerson4); // null (poll returns null for empty)

        // What happens when using remove() on an empty queue?
        try {
            ticketQueue.remove(); // Throws NoSuchElementException!
        } catch (java.util.NoSuchElementException e) {
            System.err.println("Caught exception: " + e.getMessage() + " when trying to remove from empty queue.");
        }
    }
}

Sorting

`Comparable`: Defining Natural Order

What is Natural Ordering?

Natural ordering is the default sort order for objects. For example, numbers sort numerically, and strings sort alphabetically. The `Comparable` interface provides a way for a class to define this "natural" ordering for its instances.

  • **Interface:** `java.lang.Comparable`
  • **Method:** `int compareTo(T other)`
  • **Contract:**
    • Returns a **negative integer** if `this` object is less than `other`.
    • Returns **zero** if `this` object is equal to `other`.
    • Returns a **positive integer** if `this` object is greater than `other`.

**Use Case:** When an object has an obvious, single, default sorting criterion (e.g., `Student` by ID, `Product` by SKU).

Coding Scenario: Sorting `Student` Objects by ID (Natural Order)


import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

// 1. Student class implements Comparable
class Student implements Comparable<Student> {
    private int studentId;
    private String name;
    private double gpa;

    public Student(int studentId, String name, double gpa) {
        this.studentId = studentId;
        this.name = name;
        this.gpa = gpa;
    }

    public int getStudentId() { return studentId; }
    public String getName() { return name; }
    public double getGpa() { return gpa; }

    @Override
    public String toString() {
        return "Student{id=" + studentId + ", name='" + name + "', gpa=" + gpa + "}";
    }

    // Define natural order: sort by studentId
    @Override
    public int compareTo(Student other) {
        return Integer.compare(this.studentId, other.studentId);
        // Alternative: return this.studentId - other.studentId; (careful with large numbers to avoid overflow)
    }
}

public class ComparableDemo {
    public static void main(String[] args) {
        List<Student> students = new ArrayList<>();
        students.add(new Student(103, "Charlie", 3.2));
        students.add(new Student(101, "Alice", 3.8));
        students.add(new Student(102, "Bob", 3.5));

        System.out.println("Before sorting (by ID):");
        students.forEach(System.out::println);

        // Sorts the list using the natural ordering defined by Student's compareTo()
        Collections.sort(students);

        System.out.println("\nAfter sorting (by ID):");
        students.forEach(System.out::println);
    }
}

`Comparator`: External & Custom Sorting

When `Comparable` Isn't Enough: `Comparator` to the Rescue!

While `Comparable` defines a single, "natural" ordering *within* a class, `Comparator` provides a way to define *external*, multiple, or highly flexible sorting criteria.

  • **Interface:** `java.util.Comparator`
  • **Method:** `int compare(T o1, T o2)`
  • **Contract:**
    • Returns a **negative integer** if `o1` is less than `o2`.
    • Returns **zero** if `o1` is equal to `o2`.
    • Returns a **positive integer** if `o1` is greater than `o2`.

**Use Cases:**

  • Sorting objects that *don't* implement `Comparable`.
  • Providing *multiple* ways to sort the same object type (e.g., sort `Student` by name, then by GPA, then by ID).
  • Sorting in reverse order.

Chaining Comparators: Building Complex Sorts

`thenComparing()`: Combining Multiple Criteria

Often, you need to sort data based on multiple attributes. For example, sort by `GPA`, and if `GPA`s are equal, then sort by `name`. The `Comparator` interface provides `thenComparing()` methods to chain multiple comparators together.

  • `Comparator.comparing(keyExtractor)`: Defines the primary sorting criterion.
  • `.thenComparing(anotherKeyExtractor)`: Defines the secondary sorting criterion (applied if primary criteria are equal).
  • `.thenComparing(anotherComparator)`: Can also chain with another `Comparator` instance.
  • `.reversed()`: Reverses the order of the current comparator chain.

This allows for highly expressive and concise multi-level sorting, making your code much cleaner than nested `if-else` blocks in a single `compare()` method.

    private String name;

        this.gpa = gpa;

    @Override
    public String toString() {

Generics

Generics: The Type Safety Revolution

The Problem Generics Solved: Runtime Errors & Verbose Casting

Before Generics (Java 5), collections stored elements as `java.lang.Object`. This led to two major issues:

  • No Type Safety at Compile Time: You could accidentally add objects of different types to the same collection. The compiler couldn't catch this mistake.
  • Verbose and Error-Prone Casting: Every time you retrieved an object from a collection, you had to manually cast it back to its original type. This was boilerplate code and if the cast was incorrect, it resulted in a `ClassCastException` at runtime.

Generics introduced type parameters, allowing you to specify the type of objects a collection (or class/method) can hold, bringing type safety to the forefront.

Coding Scenario: Pre-Generics Fragility vs. Generics Safety


import java.util.ArrayList;
import java.util.List;

public class GenericsProblemSolutionDemo {
    public static void main(String[] args) {
        // --- SCENARIO 1: Pre-Generics (Java 1.4 and earlier) ---
        // Compiler treats everything as Object, no type-checking for elements
        List rawList = new ArrayList();
        rawList.add("Hello"); // Adds a String
        rawList.add(123); // Accidentally adds an Integer

        System.out.println("--- Pre-Generics (Fragile) ---");
        for (Object item : rawList) {
            if (item instanceof String) {
                String s = (String) item;
                System.out.println("String: " + s);
            } else if (item instanceof Integer) {
                Integer i = (Integer) item;
                System.out.println("Integer: " + i);
            }
        }

        // What if we incorrectly assume all elements are String?
        try {
            String firstString = (String) rawList.get(0); // OK
            String secondString = (String) rawList.get(1); // ClassCastException at runtime!
            System.out.println("No error here yet for secondString: " + secondString);
        } catch (ClassCastException e) {
            System.err.println("Runtime Error (Pre-Generics): " + e.getClass().getSimpleName() + " - " + e.getMessage());
            System.err.println("This error should have been caught by the compiler!");
        }

        System.out.println("\n--- With Generics (Safe & Clean) ---");
        // --- SCENARIO 2: With Generics (Java 5+) ---
        // Compiler enforces type-checking at compile time
        List<String> stringList = new ArrayList<>();
        stringList.add("World");
        // stringList.add(456); // Compile-time Error! (Uncomment to see)

        // No need for explicit casting when retrieving
        String greeting = stringList.get(0);
        System.out.println("Greeting: " + greeting);
        // If we try to cast to Integer, compiler stops us:
        // Integer num = (Integer) stringList.get(0); // Compile-time Error!

        // Benefits:
        System.out.println(" - Compile-time type checking for safety.");
        System.out.println(" - No need for explicit casting, cleaner code.");
        System.out.println(" - Write more reusable and robust code.");
    }
}

Generic Classes & Methods: Building Flexible Structures

Generic Classes: Design for Reusability

A generic class is a class that can operate on types specified by the user. It allows you to create classes that work with different types of objects while maintaining compile-time type safety.

  • **Syntax:** `class MyClass {... }` where `T`, `U` are type parameters.
  • **Benefits:** Avoids code duplication, works with specific types while ensuring safety.
  • **Example:** `ArrayList`, `HashMap`.

Generic Methods: Type-Safe Algorithms

Generic methods are methods that introduce their own type parameters. This allows them to be written once and used for different types of arguments.

  • **Syntax:** `public T myGenericMethod(T arg) {... }`
  • **Placement:** The type parameter `T` is declared *before* the return type.
  • **Benefits:** Implement algorithms that are applicable to different types, reducing code redundancy.

Coding Scenario: Custom Generic `Pair` Class and Generic `printArray` Method


import java.util.Arrays;

// 1. Generic Class: A simple Pair to hold two objects of potentially different types
class Pair<K, V> {
    private K key;
    private V value;

    public Pair(K key, V value) {
        this.key = key;
        this.value = value;
    }

    public K getKey() { return key; }
    public V getValue() { return value; }

    @Override
    public String toString() {
        return "Pair{" + key + "=" + value + '}';
    }
}

public class GenericClassMethodDemo {

    // 2. Generic Method: A method to print elements of any array type
    public static <T> void printArray(T[] array) {
        System.out.println("Printing array of type " + array.getClass().getComponentType().getSimpleName() + ":");
        for (T element : array) {
            System.out.print(element + " ");
        }
        System.out.println();
    }

    public static void main(String[] args) {
        // Using the Generic Pair class
        Pair<String, Integer> personAge = new Pair<>("Alice", 30);
        Pair<Double, String> coordinates = new Pair<>(10.5, "Latitude");
        Pair<Boolean, Boolean> flags = new Pair<>(true, false);

        System.out.println(personAge);
        System.out.println(coordinates);
        System.out.println(flags);

        // Using the Generic printArray method
        Integer[] intArray = {1, 2, 3, 4, 5};
        Double[] doubleArray = {1.1, 2.2, 3.3};
        String[] stringArray = {"Hello", "World", "Generics"};

        printArray(intArray);
        printArray(doubleArray);
        printArray(stringArray);
    }
}

Bounded Types & Wildcards: Producers (`? extends T`)

Bounding Types: Adding Constraints to Generics

Sometimes you want to restrict the types that can be used as type arguments in a generic class or method. Bounded type parameters allow you to do this.

  • **Syntax:** `` or ``
  • **Meaning:** `T` can be `SomeClass` itself, or any class that extends `SomeClass` (or implements the specified interfaces).
  • **Use Case:** Ensures that a generic type `T` has certain methods available (e.g., if `T extends Number`, you know `T` will have `intValue()`, `doubleValue()`, etc.).

The `? extends T` Wildcard: The "Producer" Role

The `? extends T` wildcard is used when you want to create a generic method or class that will *read* (produce) elements from a collection of `T` or a subtype of `T`.

  • Meaning: `List` means a list of `Number` or any subclass of `Number` (e.g., `Integer`, `Double`, `Float`).
  • Read/Producer Role: You can safely *read* elements from such a list as `Number` (or `T`).
  • Write Restriction: You **cannot** *add* any elements (except `null`) to a `List`. Why? Because the compiler doesn't know the exact subtype (is it `List`? `List`?). Adding an `Integer` to a `List` would be unsafe.

Trainer Note: This is a key part of the **PECS principle**: "Producer-Extends, Consumer-Super". Remember, if you only *read* from a generic collection, use `? extends T`.

Coding Scenario: Summing Numbers from Various Numeric Lists


import java.util.ArrayList;
import java.util.List;

public class BoundedTypeAndExtendsWildcardDemo {

    // Method to sum numbers from a list of any type that extends Number
    // Uses <? extends Number> because it's only PRODUCING (reading) numbers from the list.
    public static double sumNumbers(List<? extends Number> numbers) {
        double sum = 0.0;
        for (Number num : numbers) { // Safely read elements as Number
            sum += num.doubleValue();
        }
        // numbers.add(new Integer(5)); // COMPILE-TIME ERROR! Cannot add to <? extends T>
        return sum;
    }

    // A generic method with a bounded type parameter
    public static <T extends Comparable<T>> T findMax(List<T> list) {
        if (list == null || list.isEmpty()) {
            return null;
        }
        T max = list.get(0);
        for (T item : list) {
            if (item.compareTo(max) > 0) {
                max = item;
            }
        }
        return max;
    }

    public static void main(String[] args) {
        List<Integer> integers = new ArrayList<>();
        integers.add(1);
        integers.add(2);
        integers.add(3);
        System.out.println("Sum of Integers: " + sumNumbers(integers));

        List<Double> doubles = new ArrayList<>();
        doubles.add(1.5);
        doubles.add(2.5);
        doubles.add(3.5);
        System.out.println("Sum of Doubles: " + sumNumbers(doubles));

        // Using findMax with a list of Integers
        System.out.println("Max Integer: " + findMax(integers));

        // List<String> strings = new ArrayList<>();
        // strings.add("hello");
        // findMax(strings); // COMPILE-TIME ERROR! String does not extend Number (for sumNumbers)
                            // or String is Comparable, but not limited by 'findMax' if it were <T extends Number>
    }
}
