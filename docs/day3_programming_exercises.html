Classes & Encapsulation

Duration: 25 min | Scope: Build a value object with validation

Create a `StudentProfile` class that keeps name, email, GPA, and shields any invalid GPA (0.0-4.0). Include factory methods and encapsulated setters.

public final class StudentProfile {
  private final String name;
  private final String email;
  private double gpa;

  private StudentProfile(String name, String email, double gpa) {
    this.name = Objects.requireNonNull(name, "name");
    this.email = Objects.requireNonNull(email, "email");
    updateGpa(gpa);
  }

  public static StudentProfile create(String name, String email, double gpa) {
    return new StudentProfile(name.trim(), email.toLowerCase(), gpa);
  }

  public void updateGpa(double gpa) {
    if (gpa < 0 || gpa > 4) {
      throw new IllegalArgumentException("GPA must be 0-4");
    }
    this.gpa = gpa;
  }
}
  • Instantiate via the static factory to centralize normalization.
  • Throw early for invalid GPA and prevent exposing setters that allow invalid state.
  • Use immutable fields + validation to keep the object stable.
Best practice: private constructor + factory Pair exercise: add `displayName()` and `toDTO()` methods

Inheritance and Method Overriding

Duration: 30 min | Scope: Model account hierarchy

Implement `Account`, `SavingsAccount`, and `CheckingAccount`. `Account` holds `balance` and defines `deposit` plus an abstract `applyMonthlyCharge`. Child classes override the charge behavior.

public abstract class Account {
  protected double balance;

  public Account(double openingBalance) {
    balance = openingBalance;
  }

  public void deposit(double amount) {
    if (amount <= 0) {
      throw new IllegalArgumentException("Deposit must be positive");
    }
    balance += amount;
  }

  public abstract void applyMonthlyCharge();
}

public class SavingsAccount extends Account {
  public SavingsAccount(double openingBalance) {
    super(openingBalance);
  }

  @Override
  public void applyMonthlyCharge() {
    balance -= 5; // base fee
  }
}
  • Favor protected helper fields for tests but avoid exposing them publicly.
  • Override `toString()` in each child to dump type and balance (demo in lecture).
  • Add `super` calls when child extends parent logic.

Interfaces & Contracts

Duration: 20 min | Scope: Create a reportable pipeline

Define a `Reportable` interface with `generateReport()` plus a default method `header()`. Implement it in a `CourseProgressReport` class and compose it via a `ReportScheduler` service.

public interface Reportable {
  default String header() {
    return \"== Learner Report ==\";  
  }

  String generateReport();
}

public class CourseProgressReport implements Reportable {
  @Override
  public String generateReport() {
    return header() + \"\\nProgress: 80%\\n\";  
  }
}
  • Use default methods to share common header formatting.
  • Explain why multiple classes can implement `Reportable` to support dashboards and email.
  • Extend the interface with a static helper `formatLine()` to reuse logic.

Static Utilities and Constants

Duration: 15 min | Scope: Provide shared helpers

Build `DateTimeUtils` with static methods `nowUtc()` and `format(DateTimeFormatter, LocalDateTime)` plus a `static final` formatter.

public final class DateTimeUtils {
  public static final DateTimeFormatter ISO_FORMAT = DateTimeFormatter.ISO_LOCAL_DATE_TIME;

  private DateTimeUtils() {}

  public static LocalDateTime nowUtc() {
    return LocalDateTime.now(ZoneOffset.UTC);
  }

  public static String format(LocalDateTime when) {
    return when.format(ISO_FORMAT);
  }
}
  • Prevent instantiation with a private constructor.
  • Share formatters as `static final` to reuse across modules.
  • Point out thread-safety of `DateTimeFormatter` in this context.
Example: add `durationSeconds()` Discuss: why not store mutable state statically?

Exception Handling & User Defined Errors

Duration: 25 min | Scope: Build custom exception layering

Implement `InsufficientBalanceException` (extends `RuntimeException`). Update `Account.withdraw` to throw it, wrap low-level exceptions, and log root causes.

public class InsufficientBalanceException extends RuntimeException {
  public InsufficientBalanceException(String message) {
    super(message);
  }
}

public void withdraw(double amount) {
  if (amount > balance) {
    throw new InsufficientBalanceException(\"Amount exceeds balance\");
  }
  balance -= amount;
}
  • Discuss checked vs unchecked and why runtime exception fits here.
  • Wrap underlying `IOException` when persisting transactions using `throw new BusinessException(\"\", io)`.
  • Always log the exception with `Logger` before rethrowing for troubleshooting.

Collections & Ordering

Duration: 30 min | Scope: Leaderboard and comparator chaining

Create `StudentScore` with `Comparable` by GPA, and build a `Leaderboard` using `TreeSet` and chained comparators to sort by GPA, then name.

record StudentScore(String name, double gpa) implements Comparable<StudentScore> {
  @Override
  public int compareTo(StudentScore that) {
    return Double.compare(that.gpa, this.gpa);
  }
}

Set<StudentScore> leaderboard = new TreeSet<>(Comparator
  .comparingDouble(StudentScore::gpa).reversed()
  .thenComparing(StudentScore::name));
  • Explain why `TreeSet` needs consistent `compareTo`/`equals`.
  • Demonstrate `thenComparing` to break ties deterministically.
  • Show how iteration order follows the comparator, perfect for leaderboard slides.
