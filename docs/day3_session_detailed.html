Part 1 · Exception Handling Fundamentals

What is an Exception?

  • Exceptions are runtime disruptions; they propagate up the call stack until handled.
  • Compile-time errors vs runtime exceptions: explain compilation halts versus runtime throws.
  • Walk through the JVM handling mechanism: exception table, stack unwinding, and the importance of stack trace reading.
// stack trace example
try {
    int result = 10 / 0;
} catch (ArithmeticException ex) {
    ex.printStackTrace(); // shows call stack + line number
}

Explain that printStackTrace() reveals the sequence of method calls; emphasize logging the message plus stack trace.

1:25 ‑ 2:05 PM

Exception Hierarchy & try‑catch control

Throwable tree & catch flow

  • Throwable splits into Error (rare, unrecoverable) and Exception.
  • Checked exceptions (IOException) must be declared/handled; unchecked (RuntimeException) propagate by default.
  • Demonstrate multi-catch and ordering; more specific first, otherwise compiler error.
try {
    Path path = Paths.get("config.csv");
    List lines = Files.readAllLines(path);
} catch (IOException | SecurityException ex) {
    log.warn("File access issue", ex);
} finally {
    System.out.println("Done with file attempt");
}

Highlight catch order: IOException is checked, SecurityException extends RuntimeException but still caught. Always release resources in finally.

Custom Exceptions, throw & throws

Example

public class InsufficientFundsException extends RuntimeException {
    public InsufficientFundsException(String message) {
        super(message);
    }
}

public void withdraw(int amount) throws InsufficientFundsException {
    if (balance < amount) {
        throw new InsufficientFundsException("Need " + (amount - balance) + " more");
    }
}

Best Practice

Use unchecked for business rules that indicate misuse (InsufficientFunds). Use checked when recovery action is expected.

Quiz: When should you declare throws in the method signature?
  • A) Only for unchecked exceptions
  • B) For any exception you catch
  • C) For checked exceptions you do not handle

Correct: C.

Try-with-resources + finally insights

Resource-safe execution

try (BufferedReader reader = Files.newBufferedReader(path)) {
    reader.lines().forEach(System.out::println);
} catch (IOException e) {
    log.error("Failed to read", e);
}

AutoCloseable guarantees close() even if exception occurs. Finally runs with resources for additional cleanup; returning inside finally still executes before return.

Debug Challenge: Open file that doesn't exist; observe suppressed exception in try-with-resources. Ask what happens if close() throws as well (look at suppressed list).

Collections Framework Foundation

Why Collections?

  • Arrays are fixed-size and lack built-in algorithms—Collections add dynamic sizing, iteration helpers, concurrency versions.
  • Framework overview: Collection root, then List/Set/Queue, with Map separately.
List<String> team = new ArrayList<>();
team.add("Alice");
team.add("Bob");

List variations

ArrayList (random access), LinkedList (insert/delete), Vector (synchronized), Stack (LIFO). Use-case: prefer ArrayList unless you need constant-time insertion/removal at both ends.

Set & Map Deep Dive (3:00‑3:40 PM)

Set behavior

HashSet uses hashing to dedupe; LinkedHashSet keeps insertion order; TreeSet sorts entries via Comparator.

Map internals

HashMap uses buckets; load factor ~0.75 triggers resize; Java 8 treeifies buckets when collisions high; LinkedHashMap keeps ordering; TreeMap sorts keys.

Map<String,Integer> phoneBook = new HashMap<>();
phoneBook.put("Alice","+91 90000");
phoneBook.putIfAbsent("Bob","+91 91000");
int attempts = phoneBook.getOrDefault("Charlie", -1);

Iteration & Sorting

  • Iterations: classic for, enhanced for, Iterator/ListIterator for removal, forEach, and intro to Streams for filtering.
  • Sorting: Comparable vs Comparator. Multi-level sorting with thenComparing.
Collections.sort(students); // Student implements Comparable<Student>
students.sort(Comparator.comparing(Student::getName)
              .thenComparing(Comparator.comparing(Student::getGpa).reversed()));

Explain: Comparable gives natural ordering; Comparator used for alternate sorts and chaining.

Utility Classes & Streams (4:05‑4:25 PM)

  • Review Collections.sort/list, Collections.reverse, Collections.shuffle, Collections.binarySearch.
  • Arrays.asList to bridge arrays and collections.
List<Integer> numbers = Arrays.asList(5,3,2,9);
Collections.sort(numbers);
Collections.reverse(numbers);
Collections.shuffle(numbers);
int idx = Collections.binarySearch(numbers, 3);

Point out binarySearch requires sorted data; mention Collections utility methods are static.

Part 3 · Generics Walkthrough

Type safety & PECS

  • Generics provide compile-time check vs raw types (List vs List<String>).
  • Generic classes: class Pair<A,B>; generic methods: static <T> void printList(List<T> list).
  • Bounded params: <T extends Number>; wildcards: , ? extends, ? super.
public class Pair<A,B> {
    private final A first;
    private final B second;
    public Pair(A first,B second){ this.first=first; this.second=second; }
    public A getFirst(){ return first; }
}

Explain class instantiation: new Pair<String,Integer>("score", 95); mention type erasure removes generics at runtime.

Generics Advanced (Wildcards & Restrictions)

  • Wildcards: read-only, producers, consumers (PECS).
  • Restrictions: no primitive type parameters, cannot instantiate new T(), generic array creation limited.
  • Type erasure removes generic info at runtime → use instanceof carefully.
public static void addIntegers(List<? super Integer> list) {
    list.add(10);
}
