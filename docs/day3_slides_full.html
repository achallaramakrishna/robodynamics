Slide 3

What is an Exception?

Runtime disruption

An exception is an object thrown when a method cannot complete normally.

It propagates up the call stack until a handler catches it.

Slide 4

Compile-time vs Runtime

Errors vs Exceptions

Compile-time errors are detected by the compiler, while runtime exceptions occur during execution.

Recognizing the difference helps you debug faster.

Slide 5

Throwable Hierarchy

Throwable, Error, Exception

Throwable splits into Error (critical JVM issues) and Exception (recoverable runtime conditions).

Exception divides into checked and unchecked branches.

Slide 6

Checked Exceptions

Compiler enforces handling

Examples include IOException and SQLException.

You must catch them or declare them with throws.

Slide 7

Unchecked Exceptions

RuntimeException family

NullPointerException and ArithmeticException extend RuntimeException.

They can be left uncaught, so guard inputs carefully.

Slide 8

try-catch Basics

Catching flow

Place risky code inside try and follow with catch blocks for matching types.

Log exceptions before recovering or rethrowing.

Slide 9

Multi-catch

Java 7 feature

Use catch (IOException | SecurityException ex) to share handling logic.

Helpful when several exceptions need identical processing.

Slide 10

finally Block

Cleanup guard

finally executes regardless of whether an exception occurred (unless System.exit interrupts).

Great place to log completion or release resources outside try-with-resources.

Slide 11

throw vs throws

Raising vs declaring

Use throw to raise a specific exception instance.

Use throws in the method signature to delegate responsibility.

Slide 12

Custom Exceptions

Domain signals

Extend RuntimeException for business rule violations (e.g., InsufficientFunds).

Include contextual data like error codes.

Slide 13

Try-with-resources

Auto cleanup

AutoCloseable resources close automatically even during exceptions.

Great for InputStream/BufferedReader stacks.

try (BufferedReader reader = Files.newBufferedReader(path)) {
    reader.lines().forEach(System.out::println);
} catch (IOException e) {
    log.error("Read failed", e);
}
Slide 14

Logging Strategy

Context-rich logs

Log the event along with the offending value and stack trace.

Wrap exceptions with more contextual ones before rethrowing.

When to log?

Log when the exception crosses a module boundary or holds actionable data.

Avoid logging the same exception multiple times.

Slide 21

Collections Intro

Why arrays fail

Arrays are fixed size with no utility methods.

Collections provide resizing, iteration, and concurrency choices.

Slide 22

Framework Architecture

Collection vs Map

Collection is the root for List, Set, Queue; Map stands beside but shares helpers.

Understanding the tree makes selecting structures easier.

Slide 23

List Interface

Ordered sequences

Lists preserve insertion order and allow duplicates.

Support index-based adds/removes.

Slide 24

ArrayList

Dynamic array

Backed by an array that grows by ~50% when full.

Fast random access and iteration.

Slide 25

LinkedList

Node chain

Doubly linked nodes support quick head/tail inserts.

Random access is linear.

Slide 26

Vector & Stack

Legacy sync

Vector synchronizes every method (slow).

Stack extends Vector; prefer Deque for LIFO today.

Slide 27

Set Interface

Uniqueness

Sets rely on equals/hashCode.

No duplicates allowed.

Slide 28

HashSet

Buckets & load factor

Uses hashing to assign buckets; load factor 0.75 triggers resize.

Collision chains degrade to trees with many entries.

Slide 29

LinkedHashSet

Insertion order

Maintains insertion order via linked entries.

Ideal for deterministic iteration.

Slide 30

TreeSet

Sorted set

Red-black tree keeps elements sorted.

Provide comparator or rely on Comparable.

Slide 31

Map Interface

Key-value pairs

Maps associate unique keys to values.

Support optional compute and merge helpers.

Slide 32

HashMap

Internal design

Buckets store entries determined by hashCode.

Resizing keeps bucket occupancy balanced.

Slide 33

LinkedHashMap

LRU caches

Tracks insertion or access order.

Override removeEldestEntry for eviction.

Slide 34

TreeMap

Sorted keys

Sorted according to natural order or comparator.

Operations cost O(log n).

Slide 35

Hashtable vs ConcurrentHashMap

Thread safety

Hashtable synchronizes each method; causes contention.

ConcurrentHashMap uses lock striping for scalability.

Slide 36

Iteration Techniques

Loops & iterators

For loops, enhanced for, Iterator/ListIterator removal.

Streams and forEach lambdas simplify iteration.

Slide 37

Streams Intro

Pipeline helpers

Map/filter/reduce pipeline example.

Avoid shared mutable state on parallel streams.

Slide 38

ArrayDeque

Deque operations

Support push/pop and offer/poll with no fixed capacity.

Replace legacy Stack for LIFO behavior.

Slide 39

HashMap Phonebook

Helper methods

Use getOrDefault to supply fallback values.

Use putIfAbsent and computeIfPresent safely update.

Slide 40

Comparable

Natural order

Override compareTo in Student to sort by GPA.

Collections.sort uses natural order by default.

Slide 41

Comparator

Alternate order

Use Comparator.comparing and thenComparing for multi-level sorts.

Supply comparators when natural order doesnt match requirements.

Slide 42

Collections Utilities

Helper methods

Collections.sort, reverse, shuffle, binarySearch operate on lists.

Binary search requires sorted data.

Slide 43

Arrays Utilities

Array helpers

Arrays.asList bridges arrays to lists.

Arrays.stream builds pipelines for primitive arrays.

Slide 44

Generics Intro

Type safety

Generics enforce compile-time type checking.

Avoid raw types to prevent runtime ClassCastException.

Slide 45

Generic Class

Pair

Store heterogeneous values with two type parameters.

Typed getters keep usage safe.

Slide 46

Generic Method

printList

Static generic method prints any List.

Callers typically rely on type inference.

Slide 47

Bounded Types

Restrict type parameter to Number or subclasses.

Allows safe usage of numeric operations.

Slide 48

Multiple Bounds

& Serializable>

Combine requirements using &.

Useful when types must satisfy multiple contracts.

Slide 49

Wildcards

?, ? extends, ? super

PECS (Producer Extends, Consumer Super) guides wildcard choice.

? extends Number for reading values, ? super Integer for writing.

Slide 50

Type Erasure

Runtime view

Generics vanish after compilation; runtime sees raw types.

Avoid instanceof parameterized types.

Slide 51

Restrictions

Limitations

Cannot instantiate type parameters or create generic arrays.

Static methods must declare their own type parameters.

Slide 52

Property Files

config.properties

Use java.util.Properties to load settings outside your code.

Store API keys, credentials, feature toggles securely.

Properties config = new Properties();
try (InputStream in = Files.newInputStream(Paths.get("config.properties"))) {
    config.load(in);
}
