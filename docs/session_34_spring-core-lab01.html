Spring Core Lab 01

Understanding Tight Coupling in Traditional Java

📘 Introduction

Before learning Spring Framework, students must understand the pain point Spring solves. In traditional Java, we create objects manually using new. That often introduces tight coupling and makes changes difficult.

  • How objects are created manually
  • What tight coupling means
  • Why dependency management becomes difficult

If students don’t feel this pain, Spring Dependency Injection will feel unnecessary.

🎯 Learning Objectives

  • Create a Maven project in Eclipse
  • Configure JDK 17 correctly
  • Write interface + implementation
  • Run the application and observe output
  • Identify tight coupling and explain why DI is needed

Step 1 – Create Maven Project

Why Maven?

Maven manages dependencies and enforces a standard project structure. Most Spring projects are Maven/Gradle based.

Action:

  1. Eclipse → File → New → Other → Maven → Maven Project
  2. Select Archetype: maven-archetype-quickstart
  3. Enter Coordinates:
    • GroupId: com.robodynamics
    • ArtifactId: core-traditional-java
    • Packaging: jar
⚙ Configure JDK / Build Path (JDK 17)

Why? If Eclipse is using an older JRE/JDK, the project may fail to compile or run.

Steps:

  1. Right-click Project → Properties
  2. Java Build Path → Libraries
  3. If JRE is missing/broken → Add Library → JRE System Library
  4. Select Workspace default JDK 17 (or installed JDK 17)
  5. Apply and Close

Tip: Eclipse → Window → Preferences → Java → Installed JREs → ensure JDK 17 is added and selected.

Step 2 – Configure pom.xml

The pom.xml defines project identity and Java version. For this lab, we set Java 17.

<project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd"> <modelVersion>4.0.0</modelVersion> <groupId>com.robodynamics</groupId> <artifactId>core-traditional-java</artifactId> <version>1.0-SNAPSHOT</version> <properties> <maven.compiler.source>17</maven.compiler.source> <maven.compiler.target>17</maven.compiler.target> <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding> </properties> </project>

If your machine has only JDK 8/11 installed, set the version accordingly OR install JDK 17.

Step 3 – Package Structure

We follow standard Java packaging. Note: artifactId may contain hyphens, but package names should not. So we use: com.robodynamics.core_traditional_java

src/main/java └── com.robodynamics.core_traditional_java ├── App.java └── service ├── MessageService.java └── EmailService.java

If you see “The declared package does not match the expected package”, your folder structure and package statement don’t match.

Step 4 – Create Interface (MessageService.java)

Why interface?

Interfaces define a contract. They allow multiple implementations (Email/SMS/WhatsApp) without changing business logic everywhere.

package com.robodynamics.core_traditional_java.service; public interface MessageService { String sendMessage(String to, String message); }

Step 5 – Create Implementation (EmailService.java)

This class implements the contract and provides actual behavior.

package com.robodynamics.core_traditional_java.service; public class EmailService implements MessageService { @Override public String sendMessage(String to, String message) { return "EMAIL SENT to " + to + " | message=" + message; } }

Later you can create: SmsService, WhatsAppService, PushNotificationService, etc.

Step 6 – Create Main Class (App.java)

This is the entry point of the program. Here we manually create dependency using new. This is where tight coupling is introduced.

package com.robodynamics.core_traditional_java; import com.robodynamics.core_traditional_java.service.EmailService; import com.robodynamics.core_traditional_java.service.MessageService; public class App { public static void main(String[] args) { // Tight Coupling: direct object creation MessageService service = new EmailService(); String result = service.sendMessage( "student@robodynamics.in", "Welcome to Spring Core!" ); System.out.println(result); } }
  • Coupling point: new EmailService()
  • If implementation changes, App.java must change
  • No runtime flexibility (cannot switch without recompiling)

Step 7 – Run Application & Verify Output

Action:

  1. Right-click App.java
  2. Run As → Java Application

Expected Output:

EMAIL SENT to student@robodynamics.in | message=Welcome to Spring Core!

🔁 Full Code Flow (What Happens When You Run?)

When you run App.java, the program follows this flow:

  1. JVM starts and loads the App class.
  2. main() method begins execution.
  3. new EmailService() creates an object in memory (heap).
  4. The object reference is assigned to service of type MessageService.
  5. service.sendMessage(...) calls the implementation in EmailService.
  6. The method returns a String message.
  7. System.out.println() prints the result in console.

Important: Even though the reference type is MessageService, the actual object is EmailService. This is runtime polymorphism.

🔄 What If We Change the Implementation?

Suppose the business says: “Stop Email. Use WhatsApp.” In traditional Java, you must change App.java.

Old:

MessageService service = new EmailService();

New (requires code change):

MessageService service = new WhatsAppService();

Result:

  • Modify source code
  • Recompile
  • Redeploy

This is the exact pain Spring solves using Dependency Injection.

🚨 Common Problems & Corrective Measures

1) Red cross on project / pom.xml error
  • Right-click project → Maven → Update Project
  • Ensure internet access (Maven downloads dependencies/plugins)
  • Try: Project → Clean
2) “JRE System Library missing”
  • Project → Properties → Java Build Path → Libraries
  • Add Library → JRE System Library → Select JDK 17
3) “Unsupported class version”
  • You compiled with newer JDK but ran with older JRE
  • Set Eclipse Installed JREs to JDK 17 and re-run
4) “Package does not match expected package”
  • Ensure file is inside correct folder path
  • Folder must match: com/robodynamics/core_traditional_java
  • Package statement must match exactly
5) “NoClassDefFoundError / Main not found”
  • Run the correct class: App.java
  • Verify public static void main(String[] args) exists

🧠 Reflection & Analysis

Question 1: Where is tight coupling introduced?

Answer: In App.java at new EmailService()

Question 2: Why is this a problem?

  • Every implementation change requires code changes
  • No runtime flexibility
  • Harder to test and maintain
